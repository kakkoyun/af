# ADR-020: mdBook User Guide Structure + Machine Index

**Status:** Accepted
**Date:** 2026-04-21

## Context

`af` currently has a `README.md` covering installation and commands, and `rustdoc`
generated from doc comments. Neither is structured for navigation by human readers
or retrieval by coding agents (RAG).

Two audiences need documentation:

1. **Human users** — need a narrative guide: quickstart, concepts, command reference
   with examples.
2. **Coding agents** — need a structured index: commands, flags, synopses, examples
   in a parseable format (JSON) so a future agent working on `af` can retrieve
   "what does `af create` accept?" without parsing Markdown.

[mdBook](https://rust-lang.github.io/mdBook/) is the standard Rust ecosystem tool
for user guides. It produces a static site from Markdown, integrates with
`cargo test` via `mdbook test` (tests code blocks), and is deployed to GitHub Pages.

## Decision

### Book location and tooling

- Root: `book/` at the repo root (not `docs/book/` — keeps `cargo doc` and mdbook
  separate, both deploy from their respective output dirs).
- `book/book.toml` — minimal config; title, authors, language.
- Build: `mdbook build book` → outputs to `book/book/` (mdBook default).
- Test: `mdbook test book` — runs all fenced Rust code blocks as doc-tests.
- Autogen: `scripts/book-gen.sh` — generates `book/src/commands/*.md` from
  `af <cmd> --help` output. Runs on `just book-gen` (already scaffolded in Lane D).

### Machine-readable index: `commands/index.json`

Generated by `scripts/book-gen.sh` alongside the Markdown pages. Schema:

```json
{
  "version": "0.1.0",
  "generated_at": "2026-04-21T00:00:00Z",
  "commands": [
    {
      "name": "create",
      "synopsis": "Create a new workstream: worktree + multiplexer + agent.",
      "flags": [
        { "name": "--from", "short": null, "takes_value": true,
          "help": "Fork from a specific branch" },
        { "name": "--remote", "short": null, "takes_value": false,
          "help": "Launch on a remote exe.dev VM" }
      ],
      "examples": [
        "af create fix-auth-bug",
        "af create --remote --agent codex ml-experiment"
      ]
    }
  ]
}
```

The JSON is committed alongside the generated Markdown. It is deterministic (sorted
by command name, flags sorted alphabetically) so diffs are stable.

### Golden test: `tests/help_snapshot.rs`

A table-driven test verifies that `af --help` and `af <cmd> --help` output match
committed snapshot files under `tests/snapshots/help/`. If the CLI output drifts,
the test fails, forcing a deliberate update to both the snapshots and the book pages.

```rust
#[test]
fn help_snapshots_match() {
    for cmd in ["", "create", "done", "list", /* ... */] {
        let output = Command::cargo_bin("af").unwrap()
            .args(if cmd.is_empty() { vec!["--help"] } else { vec![cmd, "--help"] })
            .output().unwrap();
        let snapshot = include_str!(concat!("snapshots/help/", cmd, ".txt"));
        assert_eq!(String::from_utf8_lossy(&output.stdout), snapshot);
    }
}
```

### CI: deploy alongside rustdoc

In `.github/workflows/docs.yml`:
1. Build rustdoc → `target/doc/`
2. Build mdBook → `book/book/`
3. Publish to GitHub Pages under:
   - `/api/` → rustdoc
   - `/guide/` → mdBook
4. Root `index.html` → two-link chooser: "API Reference" and "User Guide".

This replaces the existing meta-refresh redirect with a proper landing page.

### `just book-gen` implementation (Lane C1)

The stub recipe in Lane D will be replaced with a real script. `scripts/book-gen.sh`:
1. Build the binary (`cargo build --release`).
2. For each subcommand: run `af <cmd> --help`, parse the output, write
   `book/src/commands/<cmd>.md` with a header, the help text in a code block, and
   curated examples from a `scripts/examples/<cmd>.md` include.
3. Write `book/src/commands/index.json` from the parsed data.

The script must be idempotent (re-running produces the same output) for CI stability.

## Consequences

- Human users get a navigable, searchable guide separate from the rustdoc API reference.
- Coding agents can retrieve command metadata from `index.json` without parsing
  Markdown (important for the "documentation is RAG fodder" goal from the sprint plan).
- `mdbook test` catches broken code examples in the book automatically.
- The golden test (`help_snapshot.rs`) prevents silent CLI UX regressions.
- Maintaining two documentation sources (book + rustdoc) adds authoring overhead.
  Mitigated by autogen: most of `commands/*.md` is generated; only the narrative
  intro sections require manual writing.
- `scripts/book-gen.sh` has a dependency on the built binary. CI must build the
  binary before running the script. This is already the case for integration tests.
